# 实验一：Kali Linux QEMU 虚拟化技术实验报告

## 一、实验目的

- 掌握在 Linux 环境中部署 QEMU 虚拟化工具的方法。
- 学习使用 QEMU 创建与管理虚拟机的基本流程。
- 完成 Linux 系统在 QEMU 环境下的安装与配置，并能够启动运行。

## 三、实验原理与基础知识

### （一）QEMU 虚拟化基本原理

QEMU 是一种开源的机器模拟器和虚拟化工具，可以模拟 CPU、内存、磁盘、网卡等硬件设备，使操作系统能够在虚拟硬件环境中运行。实验中使用的 `qemu-system-x86_64` 用于启动 x86_64 架构的虚拟机，并通过命令行参数指定虚拟机所需的硬件资源。

QEMU 既可以进行纯软件模拟，也可以在支持硬件虚拟化的环境中结合 KVM 等加速机制提升运行效率。在 macOS（ARM 架构）环境下，QEMU 默认通过软件模拟方式运行 x86_64 虚拟机。本实验的重点是掌握 QEMU 的基本安装、镜像使用和虚拟机启动方式。

### （二）Kali Linux QEMU 镜像

Kali Linux 官方提供了面向 QEMU 的预构建镜像，常见格式为 `qcow2`（QEMU Copy-On-Write 2）。该格式支持按需分配空间、快照等特性，适合实验环境中快速部署虚拟机。与从 ISO 镜像完整安装系统相比，使用预构建的 QEMU 磁盘镜像可以省去安装步骤，直接启动进入实验环境。

### （三）虚拟机启动参数说明

实验中使用的启动命令如下：

```bash
qemu-system-x86_64 -hda kali-linux-2025.4-qemu-amd64.qcow2 -m 2048 -smp 2 -net nic -net user
```

各参数含义如下：

- **`-hda`**：指定虚拟机的硬盘镜像文件，本实验中指向 Kali Linux 的 `qcow2` 镜像，虚拟机从该镜像读取系统文件并启动。
- **`-m 2048`**：为虚拟机分配 2048 MB（2 GB）内存。内存过小可能导致系统启动缓慢或图形界面卡顿，需要结合主机硬件资源合理设置。
- **`-smp 2`**：为虚拟机分配 2 个虚拟 CPU 核心。增加核心数可以提升多任务运行效率，但也会占用更多宿主机资源。
- **`-net nic -net user`**：`-net nic` 创建虚拟网卡，`-net user` 使用 QEMU 的用户模式网络（SLIRP）。该方式配置简单，虚拟机可通过 NAT 访问外部网络，适合实验场景。

## 四、实验内容

### （一）QEMU 环境搭建

#### 1. 更新系统软件包列表

执行以下命令更新系统软件包列表，从软件源获取最新的软件包索引：

```bash
sudo apt update
```

执行时出现 `NO_PUBKEY` 提示，说明 Kali 软件源的签名公钥缺失或本地密钥信息已过期，系统无法验证软件源索引的可信性。该问题的原因是实验环境中预置的仓库密钥过期或不完整，而 Kali 软件仓库的签名密钥已更新。

解决方法：手动下载并导入最新的 Kali 官方仓库 GPG 公钥，更新本地密钥环后再次执行 `sudo apt update`。重新更新后，软件包列表正常刷新。

![Kali 软件源公钥错误](image/error.png)

如图所示，系统在更新软件包列表时提示缺少公钥，导致仓库签名无法验证。这说明当前环境的软件源配置并非完全可用，如果不先处理该问题，后续安装 QEMU 时可能无法正常获取软件包，或者继续使用旧的软件包索引。

![更新软件包列表成功](image/1.1.png)

导入 Kali 官方仓库 GPG 公钥后再次执行更新命令，终端能够正常从软件源获取软件包索引，未再出现签名验证失败提示。该结果说明软件源可信校验已恢复，实验环境可以继续进行软件安装。

#### 2. 安装 QEMU 组件

```bash
sudo apt install qemu-system-x86
```

该命令安装 x86 架构相关的 QEMU 系统模拟器组件，包括 `qemu-system-x86_64` 等可执行程序，用于启动 x86_64 虚拟机。

![安装 QEMU 组件](image/1.2.png)

截图显示 QEMU 相关软件包已进入安装流程并完成配置，说明系统已经成功安装 x86 架构虚拟化所需组件。完成此步骤后，主机具备使用 `qemu-system-x86_64` 启动 Kali 虚拟机的基础条件。

#### 3. 验证安装结果

```bash
qemu-system-x86_64 --version
```

通过查看版本信息确认 QEMU 已正确安装，并验证版本是否满足 QEMU 7.2+ 的实验要求。

![验证 QEMU 版本](image/1.3.png)

截图中终端输出了 QEMU 的版本信息，说明 `qemu-system-x86_64` 命令可以被系统正确识别并执行。版本验证不仅证明安装成功，也用于确认当前 QEMU 版本满足实验手册中 QEMU 7.2+ 的要求。

### （二）Kali Linux 虚拟机创建与启动

#### 1. 下载 QEMU 磁盘镜像

从中山大学开源镜像站下载 Kali Linux 2025.4 QEMU 镜像，以提高下载速度：

```bash
wget https://mirror.sysu.edu.cn/kali-images/kali-2025.4/kali-linux-2025.4-qemu-amd64.7z
```

下载完成后获得 `.7z` 压缩包，其中包含可直接用于 QEMU 启动的 `qcow2` 磁盘镜像。

![下载 Kali QEMU 镜像](image/2.1.png)

截图记录了 Kali QEMU 镜像压缩包的下载过程，可以看到镜像文件正在从指定镜像站获取。该步骤的关键结果是获得 `.7z` 格式的预构建虚拟机镜像压缩包，为后续解压并启动 Kali Linux 提供系统磁盘文件。

#### 2. 解压磁盘镜像

```bash
7z x kali-linux-2025.4-qemu-amd64.7z
```

解压后得到 `kali-linux-2025.4-qemu-amd64.qcow2` 磁盘文件，该文件是已预装 Kali Linux 系统的虚拟硬盘，后续通过 `-hda` 参数指定即可启动虚拟机。

![解压 Kali QEMU 镜像](image/2.2.png)

截图显示 `.7z` 压缩包已经被成功解压，生成了 QEMU 可直接使用的磁盘镜像文件。该文件相当于虚拟机的硬盘，其中已经包含 Kali Linux 系统，因此后续不需要再从 ISO 镜像进行完整安装。

#### 3. 启动虚拟机

```bash
qemu-system-x86_64 -hda kali-linux-2025.4-qemu-amd64.qcow2 -m 2048 -smp 2 -net nic -net user
```

使用已解压的 `qcow2` 镜像启动 Kali Linux 虚拟机，分配 2 GB 内存和 2 个虚拟 CPU 核心，并配置用户模式网络。启动后可进入 Kali Linux 桌面环境。

![启动 Kali Linux 虚拟机](image/2.3.png)

截图表明 QEMU 已经根据指定的 `qcow2` 镜像启动 Kali Linux 虚拟机，并进入系统运行界面。该结果说明磁盘镜像路径、内存参数、CPU 参数和网络参数均配置正确，虚拟化环境搭建完成。

## 五、实验问题分析

实验中遇到的主要问题是执行 `sudo apt update` 时出现 `NO_PUBKEY` 错误，导致软件源索引更新失败。

**原因分析**：该问题的本质是本地 GPG 密钥环中缺少 Kali 官方仓库的有效签名公钥，软件包管理器（APT）无法验证软件源索引的可信性。实验环境中预置的仓库密钥已过期或不完整，而 Kali 软件仓库的签名密钥已更新，导致签名验证失败。

**解决方法**：手动下载并导入最新的 Kali 官方仓库 GPG 公钥，更新本地密钥环。导入完成后重新执行 `sudo apt update`，软件源签名校验恢复正常，后续 QEMU 安装流程得以继续。

**启示**：在 Linux 环境中，软件源签名校验是软件包管理的重要安全机制。任何密钥缺失或过期都可能导致软件安装受阻，不能简单跳过或忽略签名验证错误，而应确保密钥配置完整且处于有效期内。

## 六、实验总结

通过本次实验，我完成了 Kali Linux QEMU 虚拟化环境的搭建与运行验证，掌握了从安装 QEMU、获取镜像、解压镜像到启动虚拟机的完整流程。

在技术层面，本次实验加深了对以下内容的理解：

- **QEMU 虚拟化机制**：QEMU 通过命令行参数描述虚拟硬件环境，`-hda` 指定磁盘镜像、`-m` 分配内存、`-smp` 设置 CPU 核心数、`-net nic -net user` 配置用户模式网络，这些参数共同定义了虚拟机的硬件规格和运行方式。
- **qcow2 镜像格式**：预构建的 qcow2 镜像基于 Copy-On-Write 技术，支持按需分配磁盘空间和快照功能，相比传统 ISO 安装方式更加高效便捷，适合实验场景中的快速部署。
- **Linux 软件包管理**：通过排查 `NO_PUBKEY` 问题，加深了对 APT 软件源签名机制和 GPG 公钥管理流程的理解，认识到软件源安全校验在系统维护中的重要性。

本次实验为后续开展网络安全、系统分析和虚拟化环境相关实验提供了可用的基础运行环境。
